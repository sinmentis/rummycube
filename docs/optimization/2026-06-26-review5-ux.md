# RummyCube — UX / Usability / Journey Review (Review 5)

**Date:** 2026-06-26 · **Reviewer:** UX Researcher (fresh pass) · **Method:** live walkthrough + 3 in-context screenshots + source read (`src/rummikub/components/`, `board.css`, `chat.css`, `lobby.css`).
**Scope:** journey friction, clarity, information scent, first-session success. Analysis only — no source touched. Already-shipped items (combo FX, coach card, opt-in hints, prominent Draw/Submit, undo/redo icons, two-click give-up, long-press pickup, drag snap/insert-push, joker-retrieve, tap-to-place, disconnect/sync cues, timeout toast, chat quick-phrases) are **not** re-litigated.

---

## TL;DR

The single highest-impact friction is **the first-turn cliff**: a brand-new player lands on a turn facing a huge empty 9×32 felt grid, a 30-second timer already ticking, and **no instruction that the core action is "drag tiles from your rack up onto the table to build a ≥30-point run/set."** The rules modal explains *what* a run/set is but never *how* to make one, the board never says "drop here," and nothing pauses the clock while they read the coach card. Fix that one moment — teach the rack→board action in-context, anchor a start target on the empty board, and relieve the first turn's timer — and first-session success jumps. The secondary theme is that the only zero-friction way to *try* the game (solo) is hidden, mislabeled "test," and then cluttered with a chat-to-nobody and a live countdown.

---

## Findings (ranked by value)

### 1. The core action — "drag a tile from your rack onto the table" — is taught nowhere — Impact: High × Effort: M
**Friction:** Both teaching surfaces skip the actual verb. The coach card (`CoachCard.jsx`) states the *goal* and the *≥30 rule* but never the gesture; How-to-play (`howToPlayContent.js`) defines Goal/Run/Set/First-meld/Jokers/Timer — all *nouns and rules*, zero *interaction*. The empty board grid carries only faint cell lines and no label.
**Why it hurts:** A first-timer reads "your first meld must total ≥30 in runs/sets," looks at 14 tiles and a vast empty green table, and has no bridge to *do* anything. The board only reveals it's a drop surface mid-drag (`.slot-valid` appears after you already guessed to drag). This is the moment people quit.
**Fix:** Add one imperative line to the coach card — *"Drag tiles from your rack up onto the table to form a run or set (≥30 points to start)."* On a still-empty board during your turn, show a faint ghost target row with *"Drop a run or set here →"* that disappears on first placement. Add a "Moving tiles" entry to How-to-play.

### 2. Promote a first-class **Play solo / Practice** path; "0 · solo test" is hidden and mislabeled — Impact: High × Effort: S
**Friction:** The only single-browser way to try the game is buried as option `0 · solo test` inside the **Number of players** dropdown (`CreateGame.jsx:75`), defaulting to **4**. The word "test" signals "throwaway / for developers," not "play by yourself / practice."
**Why it hurts:** The fastest path to "I get it" is trying the game without recruiting a friend. A normal evaluator never thinks to set *players = 0*, so they either bounce or create a 4-player room that never fills. The label actively discourages the one low-commitment on-ramp.
**Fix:** Surface a primary **"Play solo"** button on the lobby hero (peer to Create/Join), routing to the same single-player game. In the dropdown, rename to **"1 · just me (practice)"**. Treat solo as a real practice mode (ties into #4, #6).

### 3. The empty 9×32 board has no "start here" scent and wastes most of the screen — Impact: High × Effort: M
**Friction:** The board is a fixed `9 rows × 32 cols` tray (`constants.js`; `GridContainer.jsx` renders every cell at `2.2vw × 7vh` → ~63vh tall, always full size even with zero tiles). Screenshot `review5-03` shows the result: a large dead felt expanse above the rack. On mobile it's worse — `repeat(32, max(8.4vw,48px))` = a ≥1536px-wide grid you must **scroll horizontally** to traverse (`board.css:1069`).
**Why it hurts:** 288 identical empty cells give no information scent about *where* play begins; the eye has nothing to anchor to. The dead space also pushes the rack and the chat gutter around without earning its footprint. On a phone, hunting for a cell across a horizontally-scrolling board is tedious.
**Fix:** Let the visible tray grow with content early-game — render a compact "table" (e.g. 1–2 rows) until tiles exist, expanding as melds are placed — or overlay a centered start affordance on the empty state. On mobile, fit the board to viewport width (wrap/zoom) so the first meld is reachable without horizontal scroll.

### 4. The turn timer keeps ticking through onboarding and over a thinking game — Impact: High × Effort: S
**Friction:** Nothing pauses the deadline while the player reads. The coach card and How-to-play modal sit *on top of* a running clock (`TurnDeadlineWatcher` → `forceEndTurn`); the only freeze is the pre-match waiting gate (`Board.jsx:413`). Screenshots `review5-02/03` show the ring already down to 8s / 4s. Default is 30s (`CreateGame.jsx:13`).
**Why it hurts:** A first-timer reading the coach card can be force-passed mid-sentence — staged work rolled back, a tile drawn — before they've understood the move. Even fluent players experience Rummikub as a *planning* game; a 30s default that can't be paused punishes the exact deliberation the game is about. In solo "practice" this is especially wrong.
**Fix:** Pause/suspend the deadline while the coach card or How-to-play modal is open, and grant the very first turn extra (or untimed) seconds. In solo/practice, default to no timer (or a generous 60–90s). Optionally surface the numeric seconds, not only the ring.

### 5. The rich manipulation gestures are completely undiscoverable — Impact: Med-High × Effort: M
**Friction:** Long-press progressive multi-pickup, drag-snap with insert/push-and-separator, drag-onto-joker retrieve, tap-to-place, keyboard placement, and even Sort runs/colours + Undo/Redo are documented nowhere in-product. Undo/Redo are icon-only (`↶ ↷`) floating at the board's right edge.
**Why it hurts:** These are the features that make the game feel good — but a new player can only find them by accident. Without multi-select they assemble a 30-point meld one tile at a time; without "sort" they hand-arrange; they never learn joker-retrieve. The product's craftsmanship is invisible at exactly the moment it would build confidence.
**Fix:** Add a compact **"Controls / Tips"** section to How-to-play (drag to place · long-press to grab a run · drag onto a joker to take it · tap a tile then tap the table · Sort to auto-arrange · Undo/Redo). Layer 1–2 just-in-time nudges (e.g. first time a 3+ run is selected, a one-time *"Tip: long-press to grab the whole run"*).

### 6. In solo, the chat panel is pure noise and the desktop reserves a 316px gutter for it — Impact: Med × Effort: S
**Friction:** `ChatPanel` renders unconditionally (`Board.jsx:748`, no player-count gate), so a 1-player game shows a "Chat / Say hi 👋" box addressed to nobody, and desktop permanently pads `padding-right: calc(300px + 16px)` (`board.css:33`) to make room for it — squeezing the board (compounds #3).
**Why it hurts:** Chatting with an empty room is confusing and faintly absurd; worse, it steals horizontal space from the felt in the mode (#2) most likely to be a newcomer's first experience.
**Fix:** Hide chat (and drop the gutter) when `numPlayers <= 1` / no other seats are filled. On desktop multiplayer, consider a collapsible chat so players can reclaim the felt; reveal it on new-message.

### 7. Inviting is fragmented and silent — the central "Waiting for players" card has no Share button — Impact: Med × Effort: S
**Friction:** On Create, the join *link* is auto-copied (`CreateGame.jsx:28`) but the user is immediately navigated to `/match/:id`, so they never see a "copied — now send it" confirmation. In the room, the prominent center modal only says **"Waiting for players · 1 of 2 joined"** (`Board.jsx:680`) with **no share control**; the actual invite (code + Copy link) is a small panel in the top-left corner (`Sidebar.jsx`).
**Why it hurts:** The host's mental model is "I made a room — how do I get my friend in?" The answer is in the corner, disconnected from the modal that's literally about waiting. People sit on "Waiting…" not realizing they must (and how to) send the link.
**Fix:** Put the **Copy invite link** button (and the code) *inside* the waiting card, with a confirmation toast. Keep the auto-copy but confirm it ("Link copied — send it to a friend to join").

### 8. Join asks for a "Room code" but the placeholder says "Enter match ID," and pasting the shared *link* fails — Impact: Med × Effort: S
**Friction:** `JoinGame.jsx` labels the field **Room code** but the placeholder reads **"Enter match ID"** (two names for one thing), and the artifact people actually receive is a full URL (`/join-match/<id>`). Pasting that URL into the code box calls `listSeats(url)` → "Match not found."
**Why it hurts:** Terminology mismatch erodes trust ("is a code the same as an ID?"), and the most natural action — paste what my friend sent me — produces a dead-end error with no recovery hint.
**Fix:** Unify on one term ("room code") everywhere. Accept a pasted full link by extracting the trailing id before lookup. If "Match not found," hint: *"Paste the room code or the full invite link."*

### 9. Landing: the disabled **Create** gives no reason, and there's no preview of the actual game — Impact: Med × Effort: S
**Friction:** With an empty username the Create button is greyed (`CreateGame.jsx:104`) with no inline explanation (screenshot `review5-01`). The hero is text-only — three bullets, no image — so a visitor never *sees* what RummyCube looks like before committing a username and a 4-player room.
**Why it hurts:** A greyed primary button with no "why" reads as broken or unexplained. And asking for identity + player count *before* showing any payoff is a high-commitment first ask for an unproven product.
**Fix:** Add helper text/aria on the disabled state (*"Enter a username to start"*) or focus the username field by default. Add a small board screenshot/loop or a one-click "Play solo" (#2) so visitors can taste the game before filling a form.

### 10. Mobile reach: a long rack→board drag, no aim for auto-place, and a chat FAB in the thumb zone — Impact: Med × Effort: M
**Friction:** Tiles live in the bottom rack; the board fills the top — so placement is a long, two-zone vertical drag across a phone, often combined with horizontal board scroll (#3). The keyboard/auto path places at `firstFreeBoardCell` (row-major first empty), so you can't *aim*. The chat FAB is pinned bottom-right (`chat.css:233`) in the primary thumb zone, near the wrapping control cluster.
**Why it hurts:** One-handed play is awkward: big drags are error-prone, "first free cell" scatters tiles where you didn't intend, and the FAB competes with Draw/Submit/Hints for the same corner.
**Fix:** Make tap-to-place target the *visible* board region (or the last-active meld) rather than the first global free cell. Offset the FAB above the rack so it clears the control row. Consider a "send to table" affordance on a selected tile for pure-tap play.

### 11. Tile suits are encoded by colour only — runs/sets are hard for colour-vision-deficient players — Impact: Med × Effort: M
**Friction:** A run is "3+ consecutive numbers in a *single colour*" and a set is "same number in *different colours*" — yet tiles distinguish suit by colour alone (red/blue/black/orange). The submit button added a ✓/✕ glyph as a non-colour channel (good), but the *tiles themselves* have no such redundancy.
**Why it hurts:** Red/orange (and sometimes blue/black) are exactly the confusable pairs for common colour-vision deficiencies. A player who can't reliably tell suits apart cannot verify their own runs/sets — a correctness problem, not a cosmetic one — and hints/validation won't compensate for misreading the board.
**Fix:** Add a redundant suit cue on each tile (a small suit glyph/letter, or a subtle pattern/corner mark), at least as an opt-in "high-contrast / colourblind" toggle. This is the one inclusive-design gap that affects core play comprehension.

## Suggested sequencing
- **Quick wins (S):** #2 Play-solo button + rename, #4 pause timer on coach/modal + relieve first turn, #6 drop solo chat/gutter, #7 share button in waiting card, #8 join terminology + link-paste, #9 disabled-Create helper.
- **Higher-leverage (M):** #1 teach the rack→board action in-context, #3 right-size the empty board, #5 Controls/Tips + JIT nudges, #10 mobile placement, #11 colourblind tiles.

**One thing first:** ship #1 + #4 together — teach the core gesture *and* stop the clock from punishing the player while they learn it. That pair most directly converts a confused first turn into a successful one.
