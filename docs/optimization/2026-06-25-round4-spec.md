# RummyCube Round-4 优化 spec（手感打磨 + 两个 bug）

> 状态：设计已与 owner 确认（2026-06-25）。本轮 6 个工作流 WS-A…WS-F。
> 其中 WS-E（挤位留缝）、WS-F（倒计时卡 0）是真 bug 且已定位根因；其余四项是交互/手感打磨。
> 约束：服务器仍是唯一权威，anti-cheat 守卫不改；动画一律 `prefers-reduced-motion` 门控并保留非动画第二通道；

## WS-A · 长按逐张取牌（B 模式，右向累积）

**现状**：`Tile.jsx` 长按 `LONG_PRESS_MS=250` 后一次性 `onLongPress(tile)`，Board 用 `boardUtil.js` 的 `contiguousGroup`
**双向**扩展选中"按住那张 + 左右相连整组"。

**目标行为（owner 确认 B）**：长按累积式取牌。按住 250ms 起拿第 1 张（按住的那张），之后**每 250ms 往右多拿 1 张**
（同 grid/row 的连续列，遇空格停）；**左侧即使相连也永不选**；松手或移动 > `MOVE_CANCEL_PX(6)` 即停。右侧取完后继续按住不再增长（空转无害）。

**方案**：
- 新纯函数 `tilesRightward(tilePositions, pressedTileId)` → 返回 `[pressedId, ...右侧连续列 id]`（列升序；HAND grid 按 playerID 隔离，board 不隔离；遇空格停）。等价于现 `contiguousGroup` 去掉左向 `unshift` 那段。`contiguousGroup` 若无其它调用方则移除/替换。
- `Tile.jsx`：把一次性 `setTimeout(onLongPress, 250)` 改成**重复 tick**。按下后 250ms 首发，之后每 `LONG_PRESS_STEP_MS=250` 一次 tick；`firedRef` 一旦置真即吞掉随后的 click；指针移动超阈值或 up/cancel/leave 时清掉计时器。
- 累积归属：Board 在长按开始时算出右向有序序列，按已 tick 的张数把前 N 张并入 `selectedTiles`（`tryOrderTiles`/既有多选写入路径）。序列耗尽后多余 tick 不再增长。
- 视觉：每张被拿起即套用 WS-B 的选中"拿起"样式。

**验收**：
- `123`（手牌/桌面同 row 连续）按住中间那张：250ms 选中它 → +250ms 含右邻 → 再 +250ms 含右二；左侧永不进选区。
- 遇空格停（右侧到空格不再增长）。
- 真拖拽（移动 >6px）仍触发 dnd-kit 拖拽、不误触长按。
- `tilesRightward` 纯单测：左侧不含、遇空格停、HAND 跨 playerID 隔离、board 不隔离、单张无右邻 → 只含自己。

**风险**：低。与既有 6px 取消阈值/firedRef 吞 click 机制共存。

## WS-B · 选中"拿起"动画

**现状**：`getTileStyle` 内联：选中 = 灰底 `#c0c0c0` + 2px 紫边 `#6416ff`，纯平、无层次。

**目标**：选中牌有"被拈起"感。

**方案**：
- 给选中牌的 `TilePreview` 加 `tile-selected` 类（className 追加，不再只靠内联底色）。
- `board.css` 定义 `.tile.tile-selected`：`transform: translateY(-6px) scale(1.04)` + 柔和投影（如 `0 8px 16px rgba(0,0,0,.35)`）+ `transition`；提升 `z-index` 避免被邻牌盖住。
- 减动门控：`transform`/`transition` 包在 `@media (prefers-reduced-motion: no-preference)`；减动用户保留静态描边/底色作为第二通道（选中态依旧可辨）。

**验收**：选中即抬起+投影、平滑过渡；减动模式下无位移但仍有清晰选中描边；`board-visual` 源断言含 `.tile-selected` 的 transform 且在 reduced-motion 门控内。

**风险**：低（纯 CSS + 一个 className）。

## WS-C · 光标换套（去掉"十字四向"移动光标）

**现状**：`getTileStyle` 牌面内层 `cursor:'move'`（四向箭头，owner 不喜欢）；外层 wrapper 已是 `grab`。

**目标**：统一"抓取"语义。

**方案**：牌面 rest 态 `cursor: grab`（张开手），拖拽中（`isDragging`）`cursor: grabbing`（握拳手）；移除内层 `move`。不可拖时 `default`（保持现状）。

**验收**：悬停可拖的牌为张开手、拖动中为握拳手；不可拖牌无抓取光标。

**风险**：极低。

## WS-D · 落点提示：只提示桌面 + 调柔

**现状**：`GridSlot` 空格在 `(isDragActive || hasSelection) && canDnD` 时套 `.slot-valid`（绿底 `rgba(120,200,130,.16)` + 2px 内环 + 12px 辉光）；`isOver` 时再叠内联实色绿 `rgba(71,179,86,.43)`。提示对**所有** droppable 空格生效，包含手牌区。

**目标**：① 只在**桌面（board）空格**给落点提示，手牌空格永不亮（无意义）；② 把提示调柔，读作"可以放这"，去掉现在那块过于刺眼的实色绿。

**方案**：
- `GridSlot` 用 `gridId` 把 `slot-valid` 与 isOver 高亮**限定到 board grid**（`gridId === BOARD_GRID_ID`）；手牌 grid 不加提示类、不加 isOver 底色。
- 重设计 `.slot-valid`：更淡填充 + 细内描边/虚线，给"轻盈可落"的邀请感（弱化辉光）。把 isOver 的内联实色绿改成独立类 `.slot-over`（比 slot-valid 稍强但仍克制），由 GridSlot 按 `isOver && board` 切换。
- 保持 tap-to-place 行为不变（空格仍是点按落子目标），仅改视觉与作用域。

**验收**：拖拽/选中时仅桌面空格亮、手牌空格不亮；提示明显比现在柔和；hover 单格有稍强但不刺眼的 over 态；tap-to-place 落子路径不变。

**风险**：低（GridSlot 作用域 + CSS）。

## WS-E · 挤位留缝（bug，难点）

**现状/复现**：桌面 `123_456`（两组间一格空隙）。把新牌 `4` 拖到 `3` 后那个空隙：目标格是空的 → `insertWithPush` 走 `free` 短路直接放下，不挤位 → `1234456`（一条连续 = invalid）。

**根因**：`insertWithPush` 只在**目标格被占**时挤位；填进"两组之间的边界空格"会把左右两组黏成一条连续 run，而看板用空格作为组边界，黏一起即非法。

**目标**：把 `4` 吸附到 `3` 后并把 `456` 整体右推、留 1 格分隔 → `1234_456`。

**方案（纯几何启发式，模块不读牌面数值/颜色，维持现契约）**：
- 在 `insertWithPush` 的 `free`（目标列全空）分支里加"桥接"判定：放完 N 张于 `[T, T+N)` 后，若 **`col(T-1)` 被占 且 `col(T+N)` 被占**（这块同时贴住了左组与右组、无分隔），则把从 `col(T+N)` 起的连续 run 整体右推（ripple，遇空格停），空出 `col(T+N)` 作为分隔 → 结果在插入块与右组间留 1 格。
- **仅**"目标格本是两组间的边界空格"触发；目标格**被占**的组内插入仍走老逻辑（保持相连，如 `1235` 插 `4`→`12345` 不留缝）；左右空隙 ≥2 不触发（本就已分隔）；右推到 `maxCol` 放不下则该方向失败、退回吸附（不产生非法黏连）。
- `resolveDropDispatch` 的"是否挤位"判定加上"桥接"分支：原本 `inBounds && 目标占用` 才 push；现在 `桥接（目标空但左右皆占）` 也路由到 push。

**验收（精确 TDD）**：
- owner 例：`123_456` drop `4`@gap → `1234_456`（4 吸附在 3 后，456 右移一格，存在分隔，`isBoardValid` 通过）。
- 组内插入不回归：`1235` drop `4`@5 列 → `12345`（无多余缝）。
- 空隙 ≥2：`123__456` drop `4`@首个空格 → `1234_456`（已有分隔，不额外推）。
- 右侧推不动（贴 `maxCol`）→ 退回吸附，不出现非法黏连。
- 既有 `insert-tiles-with-push` / `resolve-drop-dispatch` 用例仍绿。

**风险**：中。纯几何、可预测、可测；已知取舍——靠"目标格是不是边界空格"区分"延长左组"与"组内插入"，不引入牌面数值判断。

## WS-F · 倒计时结束有时不强制抽牌、卡在 0（bug）

**现状/复现**：回合倒计时到 0，有时不强制抽牌，UI 卡在 0 干等。

**根因（高把握）**：服务器跑在 Azure VM、浏览器在 owner 笔记本——**两台机器墙钟不同步**。`TurnDeadlineWatcher` 到点只 `forceEndTurn()` **一次**，随即 `firedRef` 锁死 + `clearInterval`。服务器守卫 `if (!G.timerExpireAt || getSecTs() < G.timerExpireAt) return INVALID_MOVE`；当笔记本时钟比 VM **快**，这唯一一次会早于服务器 deadline 被拒，而 watcher **永不重试** → 卡 0。"有时"对应时钟偏移的方向与大小（也涵盖偶发丢包/重连期丢一次 move 的情况）。`getSecTs()` 实为毫秒（`new Date().getTime()`）。

**目标**：到点必定推进回合（强制抽牌/结算），对时钟偏移与偶发丢失鲁棒。

**方案（纯客户端；服务器 deadline 守卫不动，仍是 anti-cheat 权威）**：
- `TurnDeadlineWatcher` 去掉"一次性锁死"：只要 `remaining <= 0` 且回合未推进，就按节流**重复** `forceEndTurn`（约 `REFIRE_INTERVAL_MS≈1500`）；`timerExpireAt` 变化（新回合）经 effect 重新 arm、自然重置。
- 加客户端 slack：到本地 deadline 后再过约 `FIRE_SLACK_MS≈500ms` 才首发，抵消小幅"客户端偏快"导致的早发被拒。UI 倒计时可仍显示 0，nudge 稍后发。
- 重复副本若落到已推进的新回合：命中新回合未来的 `timerExpireAt` → 被 deadline 守卫当场 INVALID_MOVE 拒掉（无害幂等）。effect cleanup 清掉旧 interval，避免跨回合泄漏。

**验收**：
- 模拟"客户端时钟快于服务器"：首发被拒后，watcher 在 `REFIRE_INTERVAL_MS` 内重试直至服务器接受、回合推进（不再卡 0）。
- 正常无偏移：到点（+slack）一次推进，行为同今。
- `timerExpireAt` 变化时旧 interval 清除、新回合重新 arm（无重复计时器叠加）。
- 服务器 `forceEndTurn` 守卫逻辑不变（pre-deadline 仍拒）。

**备选（不采纳）**：改服务器下发"剩余毫秒"而非绝对时间戳来根除时钟偏移——改动面更大、风险更高；本轮取轻量的"重试 + slack"。

**风险**：低-中。纯客户端；需测重复发的幂等性与跨回合不泄漏。

## 全局约束 / 测试基线

- 服务器权威与 anti-cheat 守卫（`forceEndTurn` deadline、move 的 `playerID===currentPlayer`、board 校验）一律不改；WS-E/WS-F 都是客户端/几何层改动。
- 所有动画 `@media (prefers-reduced-motion: no-preference)` 门控并保留非动画第二通道（描边/字形）。
- 代码/注释/测试/commit 英文；Conventional Commits + `Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>`。
- 测试分层沿用现状：纯 jest（`tilesRightward`、`insertWithPush` 桥接、`resolveDropDispatch` 桥接分支）、CSS 源断言（`.tile-selected` transform+门控、`.slot-valid`/`.slot-over` 调柔、board-only 作用域、cursor）、RTL/计时器（`TurnDeadlineWatcher` 重试+slack 用假时钟）。全量 `npx jest` 保持全绿 + `npm run build` 无新 `console.log` + `node src/server.js` 启动 `/games == ["RummyCube"]`。
- 推荐顺序：先两个 bug（WS-E、WS-F，价值最高），再手感四项（WS-A→B→C→D）。WS-B 与 WS-A 视觉耦合（拿起样式），可一并落。
