# 终稿:RummyCube 网游化互动重做 · 专家团队综合结论

> **状态**:综合结论(主持人收敛)。五位专家独立报告见同目录 `01`–`05`,共享简报见 `00-brief.md`。
> **命题**:把"更直接的玩家互动"以**网游思路**做进 Web 版,**不要积分制**,且不破坏核心(共享桌面重组)。经典模式永久保留为默认。

## 二、关键分歧与如何收敛

分歧只有一个真焦点:**A(垃圾牌攻击)到底该不该当主角?**

- 挺 A:Game Designer(它是技巧驱动的引擎)、Product(改动最小、最快验证)、Backend(架构最省、回合制原生)。
- 疑 A:Trend(老套换皮)、UX(裸垃圾牌最劝退安静受众)。

**收敛的钥匙 = Game Designer 的重构 + Backend 的约束,正好同时化解两边的疑虑:**

1. **垃圾牌不是"乱丢的废牌",而是**:由**连击(打得好)铸造** → **预告一回合**(对手提前看到"下回合来 N 张")→ **自动瞄准当前领先者**(手牌最少者)→ 内容是**大概率可融入桌面的牌**。这把 UX 最怕的三件事(突袭、针对个人、毁掉成果且无从回应)全部反掉:**可预期、打桌面不打人、可挽回**。
2. **"防御 = 把来牌融进共享桌面"**:这恰恰**复用了 rummy 独有的自整理共享桌面**——于是它**不是**俄罗斯方块换皮(Trend 的核心顾虑),而是该品类**无法被抄**的角度。攻防都只是"往公共桌面放牌"。
3. **Backend 的硬约束顺手解决可行性**:Game Designer 原案里的"实时反应窗口"被 Backend 判为逆框架(需外接调度、Tunnel 下高延迟不公平)。**改法极其优雅——把"反应窗口"换成"你的下一个回合"**:来牌在你**下回合**正常结算时呈现,你能融入就抵消(并回弹一点反向压力),没融入的才粘到手牌。无需任何服务器时钟,完全落在 boardgame.io 的回合模型内。这同时满足 UX 的"预告 + 延迟 = 有张力但不残忍"。

于是**裸 A 与温和 B/D 的对立消解**:主选是"被正确设计的 A",其内核来自 B(武器化共享桌面),公平性来自 UX,可行性来自 Backend,最小范围来自 Product。**D(2v2)不被丢弃,而是作为"温和上手"的备选第二模式。**

## 三、收敛结论:1 主 1 备

- **主选 = 对抗模式 / Brawl(压制)**:连击驱动、预告、自动瞄准领先者的"压力牌",**靠把来牌融入共享桌面来防御**。回合制、你的回合桌面冻结、复用现有软时钟与连击档,经典模式仍是默认。
- **备选 = 2v2 组队 / Teams**:Trend 与 UX 的首选,Backend 判定便宜(playerView + 计分 + 分队,纯回合制),"惬意竞技"的暖场上手;将来可承载"队 vs 队"的压力对抗。
- **否决/搁置**:**C(实时抢牌市场)**——Backend 判为"线上系统陷阱"(逆框架、需自研实时网络、Tunnel 不友好),UX 判为"拼手速不拼脑、歧视慢速/老年/手机用户"。**E(排位/赛季)**——仅作**异步框架**保留(房间复用、再来一局现在就做;排位/MMR 需要 boardgame.io 没有的账号+评分库,个人维护吃不消,延后)。

# 规格 · 主选:对抗模式 / Brawl(压制)

**一句话钩子**:把"多人单机"的 rummy 变成直接对抗——你打得越漂亮,压力就越大地压向当前的领先者,而对手唯一的防御就是把你砸过去的牌**漂亮地融进公共桌面**。

**新系统(纯数字,无任何实体组件)**
- **压力 = 连击的副产品**:复用现有服务器端连击档(NICE / COMBO / ON FIRE)。本回合达到某档 → 铸造 N 张"压力牌"(N 随档位,**每回合封顶约 2–3**)。
- **自动瞄准**:压力**自动指向当前领先者**(手牌最少者)。v1 **不开放玩家手动选目标**(从根上杜绝造王/群殴)。
- **预告**:目标**立刻**看到"下回合来 N 张"(无突袭)。
- **来牌是可融入的**:服务器尽量挑**当前桌面/其手牌大概率能用上的牌**作为压力牌(而非纯废牌)。
- **防御 = 吸收**:在目标的**下一个回合**,压力牌呈现;若本回合把它**融进共享桌面**→ 抵消,并回弹少量反向压力;没吸收的才**粘到手牌**(于是离"清空手牌"的胜利更远)。
- **社交**:复用已有聊天气泡,做**只正向的预设表情**(被打可回"😅/👏",**禁止攻击后嘲讽**);加**"再来一局"**按钮。

**规则增量 vs 基础(逐步)**
1. **开房时切换** Brawl(经典模式仍是默认与新手首玩)。2–4 人,可异步。
2. 照经典 Rummikub 出牌:顺子/组、**起手 ≥30 不变**、自由重组共享桌面、**先清空手牌者胜不变**。
3. 你的回合若达连击档 → 铸造压力 → 自动指向领先者 → 该玩家收到"下回合来 N 张"预告。
4. 轮到被指玩家:压力牌随其手牌一起呈现;**融进桌面则抵消 + 反弹**,**未融入则并入其手牌**。
5. **回合结算时注入伤害**(见下"完整性要点"),**你的回合桌面冻结**,沿用现有软时钟。

**对起手 / 计分 / 胜负的改动**
- 起手 ≥30:**不变**。
- 计分:**无积分制、无计分垫**(Owner 明确否决)。
- 胜负:**仍是"先清空手牌者胜"**。唯一变化是"未吸收的压力牌会让你手牌变多"——**手牌越厚 = 离胜利越远**,天然复用既有胜负逻辑,**零新增胜负代码**。

**人数与时长**:2–4 人;单局略长 + "再来一局"循环把一局变成一坐(Product 的会话时长论点)。**异步可行**:压力作为推送效果,在对手回来时落地(Backend)。

**设计论点(为何增加互动 / 游戏性)**
- **攻击即谜题**:压力由"打得好"铸造、由"把来牌融进桌面"防御——技巧表达,**不靠运气、不是外挂式道具**,不会让人觉得卑鄙或随机。
- **自带橡皮筋**:自动瞄准领先者 = 领先就是 debuff,全场合力压住最接近胜利的人(Tetris 99 的 KO 逻辑),**自纠正跑标**。
- **公平且可挽回**:预告 + 下回合吸收 = 有张力无残忍(UX);打桌面不打人,杜绝造王/群殴。
- **无法被抄**:复用 rummy 独有的共享自整理桌面作为原生 PvP 界面(Trend);别家在线 rummy 都还是"平行单机"。
- **留存**:制造"你砸我,再来一局"的恩怨循环——小游戏最便宜的回访引擎(Product)。

**Top 3 风险(合并五位专家)**
1. **跑标 / 滚雪球**(因为技巧→攻击,更尖锐)。缓解:自动瞄准领先者、压力每回合封顶、**被吸收的来牌反而奖励防守方**,让落后的高手骑着攻击爬回来;设"伤害地板"(不被一套连击直接打死)。
2. **新手挫败 / 劝退安静受众**。缓解:**严格 opt-in,经典模式永久默认**;预告 + 可吸收 + 打桌面不打人 + 只正向表情 + 连败保护;尊重 reduce-motion。
3. **流动性(实时 PvP 需要在线人)**。缓解:**朋友优先**(自带对手)、**异步**(回合结算的推送效果等对手回来再落地)、**扩展现有 solo/bot 兜底**;**坚决避开**需要并发人数与实时网络的部分(C、反应窗口)。

**数字 / 网游可行性要点(Backend,实施时必须遵守)**
- **回合结算时注入伤害**(`onTurnEnd` / `applyValidMove`),**绝不做成回合内可撤销的招**——否则攻击者看到结果后可 `undo` 掉自己的攻击。
- 你在**对手回合**收到的伤害不在你的 undo 栈快照里 → **吃到的垃圾牌无法被撤销**(既有 undo 设计天然成立)。
- **掉线逃伤无效**:沿用既有 `forfeitSeat` 在服务器侧结算逃跑者手牌,伤害作为 pending 持久化。
- **不要实时反应窗口**:用"下一个回合"作为吸收时机,完全落在回合模型内,无需外接时钟、不受 Tunnel 延迟影响。

**首测开放问题**
- 连击→压力的**转化率与封顶**:会滚雪球还是会哑火?
- "大概率可融入"的来牌**服务器如何挑**?吸收成功率多高才算公平而非惩罚?
- **自动瞄准领先者**在 4 人桌是否仍有"群殴感"(即便是系统路由)?2 人桌(无目标选择)手感如何?
- "下回合再吸收"感觉像**有掌控**,还是延迟让它显得**软弱无牙**?
- **异步**下,一个 pending 攻击能挂多久才不显得过时?
- **bot 兜底**的质量门槛(rummy 分支因子大,真 bot 是独立工程)。

# 规格 · 备选:2v2 组队 / Teams

**一句话钩子**:把对抗变成"我们 vs 他们"——共享目标与隐藏协作,在一张依旧稳定的桌面上,用结盟而非攻击制造张力。

**新系统**:开房分队(2v2);队友之间**放宽 `playerView` 的隐藏**(可见彼此手牌或部分信息,做隐藏协作);胜负/计分按队结算。纯回合制,**无新基础设施**(Backend 判定便宜)。

**规则增量**:基础规则不变;**队伍任一方先清空(或队伍合计先清空)即胜**(待测两种口径);可选"队伍共享一条进度/超级条"(参考 Tetris Effect: Connected),但须防"一人扛、一人挂机"的搭便车。

**对起手/胜负**:起手 ≥30 不变;胜负由个人改为**按队**。

**人数与时长**:固定 4 人(2v2);时长接近经典 4 人局。

**设计论点**:UX/Trend 的"惬意竞技"甜点区——**最低挫败、最高暖意**,通过结盟与读搭档制造互动,而不动安静的桌面解谜内核。是把安静受众引向"有互动"的**温和上手坡**,也是 Brawl 之外的第二条腿。

**Top 3 风险**:① 搭便车(一人扛)→ 让两名队友都有不可或缺的贡献点;② 4 人协调的流动性(要凑齐 4 人)→ 朋友优先 + bot 队友兜底;③ 隐藏信息泄露/串通边界 → 用预设协作信号而非自由文字。

## 四、范围与先后(非排期,遵守"仅出规格")

- **先(MVP 验证一个命题:直接互动是否让对局更长、更愿意"再来一局")**:Brawl 的最小切片 = 连击→压力(自动瞄准)+ 下回合吸收 + 再来一局按钮;经典模式默认;朋友优先 + bot 兜底。
- **后(验证后再说)**:2v2 组队(D)、技巧型数字道具(B 的"你的回合"切片:冻结一组 / 偷看牌池——**不含**实时反应窗口)、异步房间复用与推送(E 的框架部分)。
- **不做**:实时抢牌市场(C)、玩家手动选目标、排位/赛季/MMR(需账号+评分库)、积分制(Owner 否决)。

## 五、一句话收尾
**主推"对抗模式 Brawl":让连击铸造、预告、自动压向领先者、靠融入共享桌面来防御的压力牌——把 rummy 的共享桌面变成它独有、别人抄不走的 PvP 界面,且零新增胜负逻辑、回合制原生、对安静受众严格 opt-in。备选 2v2 组队作温和上手;实时抢牌(C)作为线上系统陷阱明确放弃。**
